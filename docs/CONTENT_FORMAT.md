# Content Format

## 1. 目标

采用一个**足够简单、便于手工录入、兼容你已有习惯**的格式。

核心约定：

- 和弦写作 `(Chord)`
- 和弦出现在其所对应歌词之前
- 网站负责自动把和弦显示到歌词上方

## 2. 文件格式

每首歌一个 `.md` 文件，当前放在 `src/data/`。

示例：

```md
---
title: 海阔天空
artist: Beyond
language: yue
originalKey: C
capo: 0
tags:
  - 粤语
  - 流行
  - Beyond
---

( C )今天我
( G )寒夜里看雪飘过
( Am )怀着冷却了的心窝
( F )飘远方
```

## 3. Frontmatter 字段

### 必填
- `title`: 歌名
- `artist`: 歌手
- `originalKey`: 原调（也是默认展示 key）

### 推荐
- `sourceKey`: 原始输入内容当前所处的 key；当 `sourceKey` 与 `originalKey` 不一致时，页面默认应自动从 `sourceKey` 转到 `originalKey` 后再展示
- `language`: 语言。当前只允许三种语义：`粤语` / `国语` / `英语`。兼容输入别名：`yue` → `粤语`，`zh` → `国语`，`en` → `英语`
- `capo`: 变调夹品位
- `tags`: 标签数组

## 4. 正文规则

### 4.1 基本规则
- `(Chord)` 视为一个和弦标记
- 和弦后的文本视为歌词内容
- 空行保留

### 4.2 支持的原始输入形态

当前项目需要同时支持两种原始输入形态：

#### 形态 A：站内标准形态
即和弦直接写在歌词中：

```text
( F )拦路雨偏似 ( C/E )雪花
```

#### 形态 B：和弦行 + 歌词括号定位形态
即上一行写和弦序列，下一行歌词中的括号只用于标记换和弦位置：

```text
E/G# F#m7
让(我)愉快荡到天(边)
```

在这种形态中：
- 歌词括号里的内容不是和弦
- 括号用于标记和弦落点
- 上一行和弦按顺序对应下一行括号位置
- **不要默认在句首额外补一个和弦**；和弦只应落在括号明确标出的点上

默认解释规则：
- 若一行和弦数量与下一行歌词括号数量一致，则按顺序一一对应
- 规范化入库时，应转换为站内标准 `(Chord)` 形态
- 对 `C / 谁(当)我觉落漠时` 这类输入，正确结果应是 `谁( C )当我觉落漠时`，而不是在句首额外再加一个 `C`
- 若数量不一致或存在歧义，不应擅自猜测，应标记并向用户确认

### 4.3 示例

```text
( F )拦路雨偏似 ( C/E )雪花
( Dm7 )饮泣的你冻吗 ( C )
```

### 4.3 支持的和弦形态
- `C`
- `Am`
- `F#maj7`
- `Bb`
- `G/B`
- `Dsus4`
- `Cdim`

## 5. 非目标

当前版本暂不优先支持：
- 重型 lead sheet 指令系统
- 多声部/多轨标记
- 每小节拍号级别编辑
- 在线富文本编辑器

## 6. 兼容策略

第一阶段以兼容你既有 `(Chord)` 习惯为第一优先。
未来若要支持导入其他格式，应通过转换器实现，而不是破坏当前源格式。

## 6.1 渲染层策略（补充）

同一份 md 源数据允许在不同设备采用不同展示策略：
- 桌面端：优先保留和弦对位与原始版面关系
- 移动端：可启用阅读优先版，减少超长单行导致的阅读困难

原则：
- 不改坏源数据
- 不丢和弦
- 不丢歌词
- 渲染层可以为移动端可读性做适度重排

## 6.2 升降号显示规则（补充）

- 升号调默认显示为升号体系；降号调默认显示为降号体系
- 默认调名体系：`C / G / D / A / E / B / F#` 用升号体系；`F / Bb / Eb / Ab / Db` 以及其他非上述调名默认用降号体系
- 用户明确提供的 `originalKey` spelling 必须被尊重；例如给的是 `Ab`，页面默认展示与 transpose 工具的 0 档都必须保持 `Ab`，不能擅自改写成 `G#`
- slash chord（如 `F#/A#`、`Bb/Db`、`Cm/Eb`）必须整体使用同一套升降号体系
- 若左侧和弦按升号体系显示，则 `/` 右侧低音也应使用升号命名
- 若左侧和弦按降号体系显示，则 `/` 右侧低音也应使用降号命名
- slash bass 的拼写还应服从整个和弦的和声语境，而不是只按根音做机械同音转换
- 例如：`Cm/Eb` 正确，`Cm/D#` 不正确；因为 `Eb` 才是 `Cm` 语境下自然成立的音名
- 不应出现左边是升号、右边却混成降号（或反之）的混写，也不应出现虽同音但和声拼写明显不合理的 slash bass

## 6.3 sourceKey / originalKey 规则（补充）

当原始输入内容的 key 与歌曲真实原调不一致时：
- `sourceKey`：记录贴入内容本身当前的 key
- `originalKey`：记录歌曲真实原调，并作为页面默认展示 key
- 页面初始渲染应自动从 `sourceKey` 转到 `originalKey`
- 用户打开页面时，默认看到的应是原调版，而不是原始贴入版

## 7. 实际入库来源约定

当前项目默认的真实入库来源是：

- 从 mac Notes app 直接复制粘贴过来的 chord sheet 原文

这意味着：

- 输入文本可能不规范
- 可能有多余空行、半角/全角混用、括号不统一、和弦与歌词间距不稳定
- 可能夹杂临时备注或 notes 风格换行

### 处理原则

1. 先保留原意，再做规范化
2. 优先修正结构问题，不随意改动音乐内容
3. 不确定之处应标注或向用户确认，不硬猜
4. 最终入库时统一整理成项目标准 markdown 文件

## 8. Alternate line 约定

若原文中出现方括号 `[]`，当前默认解释为 alternate line。

例如：

```text
仍然获(得)一点感动 [仍然值得一点感动]
```

当前入库策略：采用**方案 1**，保留为紧随主歌词后的可读注释行：

```text
仍然获( Ab )得一点感动
[alt] 仍然值得一点感动
```

原则：
- 不丢信息
- 不强行并入主歌词行
- 暂不做更重的结构化设计

## 9. 纯音乐 / 无歌词括号约定

若原文中出现 `() () ()` 这类没有对应歌词内容的括号，默认解释为：
- 纯音乐段
- instrumental
- intro / outro / solo / break 等无歌词段落
- 或无歌词的和弦占位

当前规范化策略：
- 保留其音乐语义
- 只要原文中存在小节线 `|`，无论是 intro / instrumental / outro，都优先保留原始和弦文本行
- 不根据下方空括号数量去强行重建版式
- 不强行伪造歌词内容
