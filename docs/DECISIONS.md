# Decisions

> 记录项目中的关键设计决策、原因、权衡与后续影响。长期项目维护时，这份文档比“只看代码”更重要。

## D-001 — 采用 Astro 作为站点框架
- 日期：2026-04-05
- 状态：Accepted
- 背景：项目是一个长期维护的个人和弦谱静态网站，需要兼顾内容管理、静态部署、后续组件交互。
- 决策：使用 Astro 作为主框架。
- 原因：
  - 静态站适配 GitHub Pages
  - 内容型项目结构清晰
  - 可按需接入 React 组件处理 transpose 等交互
- 影响：
  - 构建简单，部署便宜
  - 后续增加内容页、标签页、搜索页都比较顺

## D-002 — 歌曲源格式继续采用 `(Chord)` 内嵌歌词格式
- 日期：2026-04-05
- 状态：Accepted
- 背景：已有大量 polygon 风格和弦谱积累，迁移成本应尽量低。
- 决策：源文本继续沿用 `(Chord)` 规则，不强制迁到更复杂的格式。
- 原因：
  - 与既有资料兼容
  - 手工录入成本低
  - 可读、可迁移、可脚本处理
- 影响：
  - 解析器实现较简单
  - 若未来要兼容其他格式，应通过导入/转换器实现

## D-003 — transpose 逻辑独立封装
- 日期：2026-04-05
- 状态：Accepted
- 背景：转调是项目核心能力，不应耦合在 UI 代码里。
- 决策：将和弦识别与 transpose 独立放在 `src/lib/chords.ts`。
- 原因：
  - 方便测试
  - 方便未来支持更多和弦写法
  - UI 与音乐逻辑分层更清楚
- 影响：
  - 未来加导入器、批处理脚本时可复用

## D-004 — 当前版本改用自定义歌曲 loader，而非 Astro content collection
- 日期：2026-04-05
- 状态：Accepted
- 背景：在当前初始化环境中，Astro content collection 未正常读到歌曲内容，影响 MVP 推进。
- 决策：暂时改为 `src/data/*.md` + 自定义 loader 读取歌曲。
- 原因：
  - 优先保证 MVP 可运行、可构建、可部署
  - 项目核心价值在于长期维护歌曲库，而不是框架特性本身
- 影响：
  - 当前实现更直接、更可控
  - 后续若需要 Astro 原生内容能力，可再评估迁回

## D-005 — 优先支持 GitHub Pages 自动部署
- 日期：2026-04-05
- 状态：Accepted
- 背景：用户已经创建 GitHub 仓库，希望长期托管与维护。
- 决策：优先接 GitHub Actions + GitHub Pages 自动部署。
- 原因：
  - 与代码仓库同平台
  - 维护成本低
  - 适合静态站点
- 影响：
  - 需要配置 Astro base/site 与 workflow
  - 后续若绑定自定义域名，需要再补部署文档
